# AgentLB Session Selection via App-Server Status (v0.3)

Date: 2026-02-28
Status: Implemented

## Summary

`agentlb` runs a background supervisor that tracks session usage/status from Codex app-server JSON-RPC, persists the latest data to `~/.agentlb/status.json`, and lets `agentlb new` choose the best session using only that file.

This revision removes the supervisor Unix socket server entirely.
It also adds explicit supervisor lifecycle commands and session listing/email targeting UX.

## Goals

1. Automatically pick the session with the most remaining usage for `agentlb new`.
2. Keep session usage fresh even when a session has no currently running app-server instance.
3. Restart crashed app-servers automatically.
4. Restart supervisor automatically on the next `agentlb` invocation if it is down.
5. Keep decision making file-based via `~/.agentlb/status.json`.

## Non-Goals

1. No supervisor HTTP API or Unix socket query API.
2. No direct `agentlb new` runtime RPC to app-servers for selection.

## Process Model

1. One background process: `agentlb-supervisor`.
2. PID file: `~/.agentlb/supervisor.pid`.
3. State file: `~/.agentlb/status.json`.
4. `agentlb` command startup behavior:
   1. Check if supervisor is alive using PID file.
   2. If missing/dead, start supervisor.
   3. Continue command flow after supervisor startup attempt.

### Supervisor CLI Model

1. `agentlb supervisor`
   1. Prints supervisor command help only.
2. `agentlb supervisor start --background`
   1. Prints current supervisor status.
   2. Starts supervisor in background if needed.
3. `agentlb supervisor restart`
   1. Stops running supervisor if present.
   2. Starts a fresh background supervisor.
4. `agentlb supervisor stop`
   1. Stops running supervisor if present.
5. Internal spawn target:
   1. `agentlb supervisor daemon` runs the foreground daemon loop.

## Supervisor Responsibilities

1. Load known session registry.
2. Maintain managed app-server processes for active sessions.
3. Subscribe to app-server JSON-RPC events and update in-memory status.
4. Persist status to `~/.agentlb/status.json` atomically.
5. Detect app-server death and restart with backoff.
6. Run periodic refresh probes for sessions without active instances.

## Inactive Session Refresh Probe (Required)

Because some sessions may not currently have app-server instances, the supervisor must force periodic status refreshes:

1. Run once at supervisor startup.
2. Run every 5 minutes (`probe_interval_sec = 300`).
3. For each session with no active app-server connection:
   1. Start an empty `codex app-server` instance in that session context.
   2. Send `initialize`.
   3. Send `account/rateLimits/read`.
   4. Keep process alive for up to 10 seconds (`probe_lifetime_sec = 10`) to receive updates.
   5. Record any `account/rateLimits/updated` and related status events.
   6. Terminate the probe instance.
4. If a session already has an active managed app-server, skip probe for that session in the cycle.

## App-Server Data Sources

The supervisor uses app-server JSON-RPC methods/events, with these as required inputs:

1. `account/rateLimits/read`
2. `account/rateLimits/updated`
3. `thread/tokenUsage/updated`
4. `thread/status/changed`
5. `turn/started`
6. `turn/completed`

## Status File Contract

Path: `~/.agentlb/status.json`

Write semantics:

1. Atomic write via temp file + rename.
2. File mode `0600`.
3. Flush at least every `status_flush_interval_sec = 2` when data changes.

Minimum schema:


## Selection Algorithm (`agentlb new`)

`agentlb new` selects a session using only `~/.agentlb/status.json`.

### Candidate filtering

1. Exclude sessions with `health != healthy`.
2. Exclude sessions whose rate-limit data is stale:
   1. `now - lastRateLimitUpdateAt > stale_sec` (default `stale_sec = 420`).
3. If all sessions are excluded, fallback logic applies (see Failure Behavior).

### Usage-left calculation

1. If primary exists: `remaining_primary = 100 - primary.usedPercent`.
2. If secondary exists: `remaining_secondary = 100 - secondary.usedPercent`.
3. If both exist: weighted blend
   1. `usageLeftPercent = (remaining_primary * usage_primary_weight_percent + remaining_secondary * usage_secondary_weight_percent) / (usage_primary_weight_percent + usage_secondary_weight_percent)`
   2. Defaults: `usage_primary_weight_percent = 60`, `usage_secondary_weight_percent = 40`.
4. If only one window exists, use that window's remaining value directly.
5. If no windows exist, use `unknown_usage_left_percent` (default `30`) and mark lower confidence.

### Scoring

1. `score = usageLeftPercent`
2. `score -= activeTurns * busy_penalty` (default `busy_penalty = 5`)
3. `score -= restart_penalty`
   1. `restart_penalty = min(restartCount * restart_penalty_per_restart, restart_penalty_cap)`
   2. Defaults: `restart_penalty_per_restart = 2`, `restart_penalty_cap = 20`
4. `score -= staleness_penalty` (when near stale threshold)
   1. `staleness_penalty = clamp((age_sec * staleness_penalty_max / stale_sec), 0, staleness_penalty_max)`
   2. Default: `staleness_penalty_max = 10`

### Tie-breakers

1. Higher `usageLeftPercent`
2. Lower `activeTurns`
3. Older `lastSelectedAt` (LRU spread)
4. Lexicographically smaller `sessionId` for deterministic final tie-break

## Restart and Backoff Policy

1. Restart dead app-server with exponential backoff + jitter.
2. Suggested sequence: `1, 2, 4, 8, 16, 30, 60` seconds, capped at 60s.
3. Crash-loop guard: if restarts in 5 minutes exceed `max_restarts_5m = 10`, mark session `unhealthy` temporarily.

## Failure Behavior

1. Supervisor dead:
   1. Next `agentlb` invocation restarts it.
2. Missing/stale/corrupt `status.json` during `agentlb new`:
   1. Wait briefly for supervisor refresh (`daemon_start_timeout_ms`, default 3000ms).
   2. Re-read `status.json`.
   3. If still unusable, create a new session.
3. Probe failure for one session does not block other sessions.

## Session Addressing UX

1. `agentlb new <target>` accepts:
   1. alias (existing behavior), or
   2. email (new behavior)
2. For email targets:
   1. resolve against existing session auth metadata.
   2. if exactly one match: run that alias.
   3. if zero matches: actionable not-found error.
   4. if multiple matches: actionable ambiguous-match error.
3. `agentlb list` prints:
   1. alias
   2. detected email
   3. absolute session path

## Configuration Defaults

1. `probe_interval_sec = 300`
2. `probe_lifetime_sec = 10`
3. `initial_probe_on_start = true`
4. `poll_interval_sec = 60`
5. `stale_sec = 420`
6. `busy_penalty = 5`
7. `unknown_usage_left_percent = 30`
8. `usage_primary_weight_percent = 60`
9. `usage_secondary_weight_percent = 40`
10. `restart_penalty_per_restart = 2`
11. `restart_penalty_cap = 20`
12. `staleness_penalty_max = 10`
13. `max_restarts_5m = 10`
14. `status_flush_interval_sec = 2`
15. `daemon_start_timeout_ms = 3000`
16. `limit_id = "codex"`

## Acceptance Criteria

1. Killing any managed app-server causes automatic restart.
2. Killing supervisor and invoking `agentlb new` restarts supervisor.
3. Supervisor startup triggers one immediate probe pass.
4. Every 5 minutes, supervisor probes sessions without active app-server instances.
5. Probe instance for each inactive session is terminated after 10 seconds max.
6. `agentlb new` picks the highest-score session using only `status.json`.
7. If `status.json` is unavailable or unusable after retry, `agentlb new` creates a new session.
8. `agentlb supervisor` prints help; `start --background` starts if needed; `restart` and `stop` work as lifecycle controls.
9. `agentlb list` prints alias/email/path with aligned columns.
10. `agentlb new <email>` resolves to the correct alias when uniquely matched.
